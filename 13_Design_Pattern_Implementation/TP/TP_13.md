## Deskripsi Program
Design pattern yang digunakan adalah State Machine (Finite State Machine), yaitu pola yang mengatur sejumlah state (status) tertentu beserta aturan perpindahan antar-state secara jelas dan terkontrol.

Pada proyek ini, pattern tersebut diterapkan di dua bagian:

Backend (backend/src/utils/orderStateMachine.js) untuk memvalidasi apakah perubahan status pesanan diperbolehkan.
Frontend (frontend/src/js/siswa/orderStateMachine.js) untuk mengatur tampilan progress tracker sesuai status pesanan.

Alur status pesanan bersifat satu arah:

ORDERED → COOKING → READY → DONE

Artinya, status hanya bisa maju ke tahap berikutnya dan tidak boleh mundur atau melompati tahap tertentu. State DONE merupakan terminal state, sehingga setelah pesanan mencapai status tersebut, tidak ada transisi lain yang dapat dilakukan.

### Implementasi Backend: 
```
'use strict';

const STATES = Object.freeze({
  ORDERED: 'ORDERED',
  COOKING: 'COOKING',
  READY:   'READY',
  DONE:    'DONE',
});

const TRANSITIONS = Object.freeze({
  ORDERED: ['COOKING'],
  COOKING: ['READY'],
  READY:   ['DONE'],
  DONE:    [],  // terminal state
});

function isValidState(state) {
  return Object.prototype.hasOwnProperty.call(TRANSITIONS, state);
}

function canTransition(from, to) {
  if (!isValidState(from) || !isValidState(to)) return false;
  return TRANSITIONS[from].includes(to);
}

function nextStates(from) {
  if (!isValidState(from)) return [];
  return [...TRANSITIONS[from]];
}

module.exports = { STATES, TRANSITIONS, isValidState, canTransition, nextStates };
```
-> Controller nya cukup tanya "boleh nggak?" ke state machine tanpa perlu tahu aturan lengkapnya. Semua logika validasi udah kepusat di satu tempat aja.

        const stepIndex    = getSteps().indexOf(step);

-> Versi frontend pakek pola `IIFE` bia state tersimpan secara privat. Fungsi `getStepStatus()` digunakan untuk menentukan apakah setiap langkah di progress tracker harus ditampilkan sebagai done, active, atau inactive.

Pattern State Machine ini punya beberapa keunggulan kalau dibandingin pakek cara biasa. Kalau nggak pakek pattern ini, aturan transisi statusnya bakal nyebar di banyak `if-else` di berbagai bagian kode. Nah kalau pakek `state machine` semua aturannya ngumpul di satu tabel `TRANSITIONS` yang gampang dibaca dan diubah kalau perlu. Terus, kasus-kasus yang kelewat kayak lompatan langsung dari `ORDERED` ke `DONE` otomatis keblock, karena transisi itu emang nggak ada di tabelnya. Error messagenya juga jadi lebih jelas karena ada fungsi `nextStates()` yang langsung ngasih tau transisi mana aja yang boleh. Terus meskipun logikanya ditulis ulang di frontend dan backend, desainnya tetep sama dan konsisten di kedua layer, tinggal disesuaiin sama kebutuhan masing-masing.
